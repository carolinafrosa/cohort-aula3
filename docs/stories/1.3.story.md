# Story 1.3 — Fuzzy Split de Nome Composto na Busca

**Status:** InReview  
**Epic:** 1 — Qualidade e UX do Chatbot  
**Story ID:** 1.3  
**Prioridade:** Alta  
**Estimativa:** S (2–4h de edição no n8n)

---

## Descrição

Quando o usuário digita um nome de projeto sem espaço entre as palavras (ex: `"viamar"` em vez de `"via mar"`), o bot passa o termo exato para o Oracle e retorna zero resultados — sem nenhuma tentativa de variação. O usuário não tem como saber que a grafia com espaço é a correta e é forçado a tentar por tentativa e erro.

Esta story corrige o problema em dois pontos:

1. **`InterpretaConsulta`** — aprimorar o prompt para instruir o LLM a identificar nomes que parecem palavras concatenadas sem espaço e retorná-los com espaçamento provável (ex: `"viamar"` → `"via mar"`).
2. **`MontaBusca`** — quando o campo `nome` contém múltiplas palavras (após o split), gerar cláusula SQL `LIKE` com `%` entre cada token, tornando a busca tolerante a variações de grafia.

**Valor:** Elimina a principal causa de "zero resultados" por erro tipográfico/grafia, reduzindo a fricção para o usuário que conhece o projeto mas não a grafia exata cadastrada no sistema.

## Acceptance Criteria

### AC-1: Aprimoramento do prompt em `InterpretaConsulta`

- [ ] O system prompt de `InterpretaConsulta` inclui instrução explícita:
  > "Se o campo `nome` parecer palavras concatenadas sem espaço (ex: 'viamar', 'viadul', 'ferrovia'), separe as palavras prováveis com espaço antes de retornar (ex: 'via mar', 'via dul', 'ferro via')."
- [ ] O LLM deve retornar o nome com espaçamento corrigido no JSON de saída
- [ ] A instrução não altera o comportamento para nomes já corretamente espaçados

### AC-2: Busca multi-token em `MontaBusca`

- [ ] Quando o campo `nome` contém 2 ou mais palavras (separadas por espaço), a cláusula SQL gerada usa `%` entre cada token:
  ```sql
  -- Exemplo: nome = "via mar"
  UPPER(TRANSLATE(NOME,...)) LIKE '%VIA%MAR%'
  
  -- Exemplo: nome = "ponte rio" 
  UPPER(TRANSLATE(NOME,...)) LIKE '%PONTE%RIO%'
  ```
- [ ] Quando o campo `nome` contém apenas 1 palavra, mantém o comportamento atual (`LIKE '%PALAVRA%'`)
- [ ] A normalização de acentos via `TRANSLATE` é aplicada igualmente em ambos os casos

### AC-3: Mensagem de "não encontrado" inalterada para zero resultados reais

- [ ] Se mesmo com o nome espaçado o resultado for zero, a mensagem de "não encontrado" é exibida normalmente
- [ ] Nenhum loop infinito ou tentativa recursiva é introduzida

## Riscos

- **Over-split:** O LLM pode separar palavras que não devem ser separadas (ex: `"sobrado"` → `"so brado"`). Mitigação: a instrução de split deve ser conservadora — apenas quando a palavra é claramente composta (≥8 caracteres sem vogais consecutivas ou padrão reconhecível).
- **Performance SQL:** O padrão `%TOKEN1%TOKEN2%` pode ser mais lento que `%PALAVRA%` simples no Oracle. Para o volume atual (projetos do governo SC), o impacto é aceitável.
