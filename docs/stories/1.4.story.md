# Story 1.4 — Detecção de Município em Consulta Composta

**Status:** InReview  
**Epic:** 1 — Qualidade e UX do Chatbot  
**Story ID:** 1.4  
**Prioridade:** Alta  
**Estimativa:** S (2–4h de edição no n8n)

---

## Descrição

Quando o usuário digita uma consulta que mistura nome do projeto e município na mesma frase (ex: `"viamar rio do sul"`), o nó `InterpretaConsulta` absorve o texto inteiro como valor do campo `nome`, ignorando que `"rio do sul"` é um município válido no banco.

A Story 1.2 implementou a resolução antecipada de filtros via `ResolveFiltros` (Groq), que mapeia termos do usuário para valores exatos de ORGAO, TIPO e MUNICIPIO. Porém, quando o nome do projeto e o município estão juntos na mesma string (sem separação clara), o LLM não consegue distinguir o que é nome e o que é município.

Esta story aprimora o prompt de `InterpretaConsulta` para instruir explicitamente o LLM a separar os tokens de uma frase composta entre os campos corretos — especialmente identificar municípios conhecidos de SC embutidos em uma consulta de projeto.

**Valor:** O usuário que sabe o município do projeto mas não a grafia exata consegue filtrar corretamente na primeira tentativa, sem precisar reformular a pergunta.

## Acceptance Criteria

### AC-1: Instrução de separação de tokens no prompt de `InterpretaConsulta`

- [ ] O system prompt inclui instrução explícita:
  > "Quando a consulta do usuário misturar nome de projeto com localização (ex: 'viamar rio do sul'), separe os tokens: o nome do projeto vai para o campo `nome` e o município vai para o campo `municipio`. Nunca coloque município dentro do campo `nome`."
- [ ] O prompt inclui exemplos concretos de separação:
  ```
  "viamar rio do sul" → {"nome": "via mar", "municipio": "Rio do Sul"}
  "ponte blumenau" → {"nome": "ponte", "municipio": "Blumenau"}
  "escola estadual florianópolis" → {"nome": "escola estadual", "municipio": "Florianópolis"}
  ```

### AC-2: LLM extrai município corretamente em consultas compostas

- [ ] Para `"viamar rio do sul"`: `municipio = "Rio do Sul"` e `nome = "via mar"` (ou variação fuzzy)
- [ ] Para `"ponte joinville"`: `municipio = "Joinville"` e `nome = "ponte"`
- [ ] Para `"projeto escola blumenau andamento"`: `municipio = "Blumenau"`, `nome = "projeto escola"`, `situacao = "Andamento"`
- [ ] Para consultas sem município: campos `municipio` permanece `null` (sem invenção)

### AC-3: Compatibilidade com `ResolveFiltros` (Story 1.2)

- [ ] O campo `municipio` extraído pelo `InterpretaConsulta` é passado para `AplicaFiltrosResolvidos` normalmente
- [ ] O `ResolveFiltros` (Groq) confirma ou normaliza o valor do município extraído contra a lista de municípios reais do banco
- [ ] A lógica de `MontaBusca` com o campo `municipio` não é alterada por esta story

## Riscos

- **Ambiguidade real:** Algumas frases genuinamente ambíguas (ex: `"campo belo"` pode ser nome de projeto ou município "Campo Belo"). Mitigação: o LLM deve preferir interpretar como município quando o token corresponder exatamente a um município conhecido da lista.
- **Conflito com Story 1.3:** As duas stories modificam `InterpretaConsulta`. Recomenda-se aplicar em sequência (1.3 primeiro, depois 1.4 no mesmo nó) ou numa única edição consolidada.
