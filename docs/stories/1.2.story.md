# Story 1.2 — Resolução Antecipada de Filtros com Valores Exatos do Banco

**Status:** InReview  
**Epic:** 1 — Qualidade e UX do Chatbot  
**Story ID:** 1.2  
**Prioridade:** Alta  
**Estimativa:** L (5–8h de edição no n8n)

---

## Descrição

Hoje o fluxo envia termos vagos do usuário (ex: "educação") diretamente ao Oracle via LIKE, causando imprecisão nos resultados e na contagem de registros. Esta story adiciona uma etapa de **resolução antecipada de filtros** antes da consulta principal: uma pré-consulta busca os valores distintos do banco (ORGAO, TIPO, MUNICIPIO), um LLM mapeia os termos do usuário para valores exatos, e esses valores exatos são persistidos no contexto e exibidos nas mensagens durante toda a sessão.

**Exemplo:**
- Usuário digita: `educação florianópolis`
- Filtros resolvidos: `Órgão: Secretaria de Estado da Educação | Município: Florianópolis`
- Consulta Oracle usa `=` em vez de `LIKE` para os campos resolvidos

**Valor:** Usuário que busca "educação" hoje recebe resultados imprecisos de múltiplos órgãos. Com a resolução antecipada, "educação" → "Secretaria de Estado da Educação" garante resultados corretos na primeira tentativa, reduzindo o ciclo de refinamento de 3–5 trocas para 1–2.

## Acceptance Criteria

### AC-1: Nó `ConsultaDistintos` (Oracle)
- [x] Nó Oracle inserido após `AplicaClassificacao` (path tipo=busca)
- [x] **3 queries separadas** (SELECT DISTINCT combinado geraria produto cartesiano inválido):
  ```sql
  -- Query 1: órgãos
  SELECT DISTINCT ORGAO FROM VW_IDENTIFICACAO_PROJETOS
  WHERE ORGAO IS NOT NULL ORDER BY ORGAO FETCH FIRST 200 ROWS ONLY

  -- Query 2: tipos
  SELECT DISTINCT TIPO FROM VW_IDENTIFICACAO_PROJETOS
  WHERE TIPO IS NOT NULL ORDER BY TIPO FETCH FIRST 100 ROWS ONLY

  -- Query 3: municípios
  SELECT DISTINCT MUNICIPIO FROM MUNICIPIO ORDER BY MUNICIPIO
  ```
- [x] Implementado como 1 nó com UNION ALL (mais simples, performance aceitável para distintos)
- [x] Retorna listas de valores válidos para os 3 campos

### AC-2: Nó `PreparaResolveFiltros` (Code)
- [x] Recebe os campos do usuário (`campos_acumulados`) e os valores distintos do Oracle
- [x] Monta o prompt para o LLM com:
  - Lista de ORGAOs válidos (até 100)
  - Lista de TIPOs válidos
  - Lista de MUNICIPIOs válidos (até 100)
  - Campos que o usuário informou (nome, municipio, orgao, tipo)
- [x] Implementado em `PreparaConsulta` (existente, modificado para ler Oracle + cachear distintos)

### AC-3: Nó `ResolveFiltros` (HTTP Request — Groq)
- [x] Chama `https://api.groq.com/openai/v1/chat/completions`
- [x] Modelo: `llama-3.3-70b-versatile`, temperature: 0, max_tokens: 300
- [x] System prompt orienta o LLM a:
  - Retornar SOMENTE JSON válido: `{"orgao": "...", "municipio": "...", "tipo": "...", "nome": "..."}`
  - Usar EXATAMENTE os valores da lista fornecida para orgao, tipo e municipio
  - Retornar `null` para campos que o usuário não mencionou
  - Preservar o campo `nome` exatamente como digitado pelo usuário (sem corrigir)
  - Se o termo não bater com nenhum valor da lista, retornar `null` para aquele campo
- [x] Implementado em `InterpretaConsulta` (existente, modificado com instruções de match exato e listas no user content)

### AC-4: Nó `AplicaFiltrosResolvidos` (Code)
- [x] Recebe a resposta do LLM e aplica ao `campos_acumulados` do contexto
- [x] Campos resolvidos substituem os vagos (ex: `orgao: "educação"` → `orgao: "Secretaria de Estado da Educação"`)
- [x] Campos com valor `null` na resposta LLM são mantidos como estavam (sem sobrescrever)
- [x] Gera `filtros_label` — string legível com os filtros ativos:
  - Formato: `🔍 *Filtros:* Município: Florianópolis | Órgão: Sec. de Educação | Situação: Em andamento`
  - Inclui apenas campos com valor real (não QUALQUER, não null)
  - Abreviações amigáveis para órgãos longos (ex: "Sec. de Educação" para "Secretaria de Estado da Educação")
- [x] Salva `filtros_label` e `campos_acumulados` atualizados no `staticData.contextos[remetente]`
- [x] Ignora parsing do LLM quando busca_original='refinar' (seleção de refinamento)

### AC-5: Modificação em `MontaBusca`
- [x] `AplicaFiltrosResolvidos` salva `campos_resolvidos: { orgao: true/false, tipo: true/false }` no contexto junto com `campos_acumulados` — esse flag é lido por `MontaBusca`
- [x] Para `ORGAO`: quando `campos_resolvidos.orgao === true`, usa `TRANSLATE(UPPER(VIP.ORGAO),...) = TRANSLATE(UPPER('valor_exato'),...)` em vez de LIKE
- [x] Para `TIPO`: mesma lógica — `=` quando `campos_resolvidos.tipo === true`, LIKE quando false
- [x] Para `MUNICIPIO`: mantém LIKE (normalização já funciona bem para municípios)
- [x] Para `NOME`: mantém LIKE (busca parcial por título é o comportamento correto)

### AC-6: Modificação em `ProcessaResultados`
- [x] Todas as mensagens de resultado (refinamento, lista de projetos, projeto único, sem resultado) incluem o `filtros_label` no início
- [x] Formato de exibição:
  ```
  🔍 *Filtros:* Município: Florianópolis | Órgão: Sec. de Educação
  
  Encontrei *41 projetos*...
  ```
- [x] `filtros_label` é lido do `staticData.contextos[remetente].filtros_label`
- [x] Se `filtros_label` estiver vazio (primeira busca sem filtros), omite a linha

## Escopo

**IN:**
- 4 nós novos no fluxo (ConsultaDistintos, PreparaResolveFiltros, ResolveFiltros, AplicaFiltrosResolvidos)
- Modificação de MontaBusca (troca LIKE por = para órgão e tipo quando resolvido)
- Modificação de ProcessaResultados (exibir filtros_label)
- Campos: ORGAO, TIPO, MUNICIPIO, NOME

**OUT:**
- Campo DESCRICAO — não existe na view VW_IDENTIFICACAO_PROJETOS com o schema atual
- Resolução de SITUACAO — já é tratada deterministicamente (1=Andamento, 2=Concluído)
- Mudanças nos nós de roteamento (Switch/IF)
- Cache dos distintos entre sessões (pré-consulta a cada busca)

## Riscos

- **Volume de distintos:** Se `SELECT DISTINCT ORGAO` retornar centenas de valores, o prompt LLM pode exceder o limite de tokens. Mitigação: `ROWNUM <= 500` e agrupar por prefixo "Secretaria de Estado da".
- **Latência:** Pré-consulta Oracle adiciona ~200–500ms por busca. Aceitável dado o ganho de precisão.
- **Falso negativo no LLM:** LLM pode retornar `null` para um órgão que existe no banco se o nome estiver muito diferente. Mitigação: manter o LIKE como fallback quando o campo resolvido for `null`.
